## 2. Core Concept: Documents as Data Primitives

**"One data abstraction. Every business object."**

A Powerhouse document is a typed, self-contained data structure defined by a **document model** — a schema that specifies its shape and the exact operations that can modify it. Under the hood, it's JSON. On the wire, it's an append-only sequence of signed operations.

**What this gives you:**

- **Schema-enforced consistency** — Operations have well-defined parameters and built-in business logic. The document state is always valid by construction, not by validation after the fact.
- **Cryptographic verification** — Every operation is signed by its author. The full history is offline-verifiable without trusting a central authority.
- **Git-like version control** — Branching, merging, undo, and history traversal come free from the append-only model. No additional development needed.
- **Storage-agnostic persistence** — The same document can live as a file, in a browser (pglite), in PostgreSQL, or on-chain. Append-only histories work in both mutable and immutable repositories.
- **AI-native by default** — Every document model automatically generates GraphQL and MCP endpoints. The same operations available to human editors are available to AI agents and integration workflows — no adapter layer.

## 3. Architecture: Reactive Document Infrastructure

**"Event-sourced. CQRS. Horizontally scalable. Local-first."**

The reactive document architecture is the infrastructure layer that turns documents from individual files into a platform-grade backend.

**Write path** — All mutations are document operations. Because documents are self-contained and independent, the write layer scales horizontally by adding nodes. There's no shared-state bottleneck; each document is its own consistency boundary.

**Read path** — Specialized processors subscribe to document update events and build denormalized read models. Store aggregated data wherever it fits: relational databases, search indexes, data warehouses, caches. Read and write concerns are fully separated (CQRS).

**Sync layer** — Documents synchronize across devices, apps, and users via their operation histories. This turns every document into a real-time collaboration channel and every editor into a collaborative application — without custom sync code.

**Result:** The same architecture pattern that powers large-scale platforms (millions of users, real-time, eventually consistent) — but local-first, open source, and running on your infrastructure.

## 4. The Platform

**"Six components. Each independent. Stronger together."**

### Connect

The document management workspace. Operators create, edit, and collaborate on business documents through a browser-based UI. **Drive apps** overlay the file system with specialized workflow interfaces — standardized best-practice processes directly available, rather than leaving operators to organize their own folder hierarchies.

### Switchboard

The API and integration layer. Exposes every document and operation as GraphQL queries/mutations and MCP tool endpoints. This is the system boundary where external applications, automation workflows, and AI agents interact with your document infrastructure. Handles authorization, sync, and multi-tenant access.

### Clint

AI-native command infrastructure. A single set of Zod-based command definitions — derived directly from your document models — produces a CLI (command mode and interactive REPL), an MCP server, and an AI agent harness. Operators use typed commands or natural language; autonomous agents use the same operations through the routine loop — a tick-based execution engine with pluggable triggers for event-driven, timer-based, or condition-based processing. One codebase, every interaction surface.

### Fusion

Custom platform builder. A Next.js-based framework for deploying branded, customer-facing applications on top of your document infrastructure. Switchboard-backed, fully customizable. Build the equivalent of a SaaS product — with the backend entirely in your hands.

### Renown

Identity and reputation layer. Verified contributor profiles and contribution tracking across your organization and partner networks. Provides the identity substrate for authorization, audit trails, and cross-organizational collaboration.

### Vetra

The platform ecosystem. Three pillars:

- **Packages** — A registry of reusable document models, editors, read models, and agent configurations. Use pre-built modules for common business domains or publish your own.
- **Hosting** — Managed cloud infrastructure with real-time sync. Production-ready deployment without hyperscaler dependency. Your data sovereignty is preserved.
- **AI Agents** — Pre-configured business process agents built on the document architecture. Deploy from the ecosystem or build custom agents with Clint.

Development tooling (Vetra Studio IDE, PH-CLI) and a learning platform (Vetra Academy) support teams building on the stack.

## 5. AI Integration Model

**"Native, Not Bolted On"**

Most enterprise AI integrations require building a translation layer between the AI and the business system. Powerhouse eliminates this by making documents the shared interface for humans and machines.

**How it works:**

- Document models generate MCP endpoints automatically. Any MCP-compatible AI agent can read and write business data through the same typed operations that human editors use.
- Clint's **agent skills** are structured process playbooks (not open-ended prompts) that encode domain expertise as step-by-step workflows. Agents follow specifications, not guesses.
- The **routine loop** enables autonomous processing: agents monitor document changes, evaluate triggers, and execute operations on a continuous cycle — with or without human oversight.
- **Provider-agnostic**: Sensitive workflows run on local open-source models (your infrastructure, your GPU). General-purpose tasks use commercial APIs. Swap providers without architectural changes.

**What stays in your control:**

- All data — documents never leave your infrastructure for model training
- All operations — every AI action is a signed document operation in the append-only history
- All logic — business rules live in document model reducers, not in prompt engineering

## 6. Technical Characteristics

**A scannable reference for architecture review.**

| Characteristic | Detail |
|---|---|
| **Data model** | Typed documents (JSON) with Zod schemas, GraphQL-based schema definition, custom scalars for business types (monetary amounts, identifiers, timestamps) |
| **Mutation model** | Append-only operation history; pure reducers (deterministic, no side effects); operation-level error handling |
| **Storage** | Pluggable adapters: filesystem, PostgreSQL, pglite (browser), in-memory, blockchain. Same document, any backend |
| **Sync** | Operation-based synchronization across devices, apps, and users. Conflict resolution via branch/merge semantics |
| **APIs** | GraphQL (Switchboard) + MCP (Switchboard & Clint). Auto-generated from document model schemas |
| **Auth** | Cryptographic operation signing; per-drive authorization; Renown identity integration |
| **Scalability pattern** | Horizontal write scaling (documents are independent consistency boundaries); CQRS read models; event-driven processors |
| **Stack** | TypeScript, React, GraphQL, Node.js 22+, Zod 4. Open source |
| **AI framework** | Mastra-based agent integration; provider-agnostic LLM backends; thread-based conversation memory; MCP tool discovery |
| **Deployment** | Self-hosted or Vetra Cloud managed hosting |
